# Managed Repo Audit Toolset Contract

**Phase:** 3  
**Module:** `operations_center.audit_toolset`  
**Related:** [Managed Repo Contract (Phase 1)](../../architecture/managed-private-project/managed-private-project_managed_repo_contract.md) · [Artifact Contract (Phase 2)](../../architecture/managed-private-project/managed-private-project_audit_artifact_contract.md)

---

## Purpose

Phase 3 defines the OperationsCenter-side toolset contract for invoking managed repo audits and discovering their artifact manifests. It connects the Phase 1 managed repo config (how to invoke) with the Phase 2 artifact contract (what the output looks like).

Phase 3 does not execute commands. It does not write manifests. It does not index artifacts. It defines and validates the shapes that Phase 6 dispatch will consume.

---

## Relationship to Phase 1 Managed Repo Config

Phase 1 (`config/managed_repos/managed-private-project.yaml`) declares:
- Which commands exist and their `command_status` evidence
- Which audit types are supported
- Where outputs appear (`output_dir`, `status_file`)
- How `run_id` is injected (`AUDIT_RUN_ID` env var)

Phase 3 reads this config and produces a validated `ManagedAuditInvocationRequest`. It enforces that only acceptable `command_status` values produce invocation contracts.

---

## Relationship to Phase 2 Artifact Contract

Phase 2 defines:
- `ManagedRunStatus` — the validated shape of `run_status.json`
- `artifact_manifest_path` — the field that links run_status to the manifest
- `is_compliant` — whether the run_status meets the full contract

Phase 3 uses Phase 2 models in its discovery layer:
- `load_run_status_entrypoint()` validates `run_status.json` against `ManagedRunStatus`
- `resolve_artifact_manifest_path()` reads `artifact_manifest_path` from the validated model

---

## Invocation Request Shape

`ManagedAuditInvocationRequest` is the validated contract object that Phase 6 dispatch will consume to actually run an audit.

| Field | Type | Description |
|-------|------|-------------|
| `repo_id` | `str` | Which managed repo to invoke |
| `audit_type` | `str` | Which audit type (e.g. `"representative"`) |
| `run_id` | `str` | UUID hex generated by OpsCenter before invocation |
| `working_directory` | `str` | Directory from which to run the command |
| `command` | `str` | The external command string (not executed here) |
| `env` | `dict[str, str]` | Env vars including `AUDIT_RUN_ID` |
| `expected_output_dir` | `str` | Base dir where run_status.json will appear post-run |
| `metadata` | `dict` | Caller-supplied metadata (channel_slug, etc.) |

The model validator enforces that `AUDIT_RUN_ID` is present in `env` and equals `run_id`.

---

## Command Resolution Rules

`resolve_invocation_request(repo_id, audit_type, run_id)` applies the following steps:

1. **Load config** — calls `load_managed_repo_config(repo_id)` from Phase 1 loader. Raises `ManagedRepoNotFoundError` if no config file exists.
2. **Verify audit capability** — checks `"audit" in config.capabilities`. Raises `ManagedRepoCapabilityError` if absent.
3. **Verify audit_type** — checks that the type is declared in `config.audit.audit_types`. Raises `ManagedAuditTypeUnsupportedError` if not found.
4. **Check command_status** — enforces the policy table below.
5. **Inject AUDIT_RUN_ID** — sets `env["AUDIT_RUN_ID"] = run_id`.
6. **Return** `ManagedAuditInvocationRequest`.

---

## Command Status Handling

| `command_status` | Default | `allow_not_yet_run=False` |
|-----------------|---------|--------------------------|
| `verified` | Allowed | Allowed |
| `not_yet_run` | Allowed | Blocked |
| `unknown` | Blocked | Blocked |
| `needs_confirmation` | Blocked | Blocked |

`unknown` and `needs_confirmation` are always blocked. Silently accepting them would mean OpsCenter might invoke commands it has no evidence for — a contract violation.

`not_yet_run` is allowed by default to support dry contract generation and pre-Phase-5 testing. Pass `allow_not_yet_run=False` when only verified commands should generate invocation requests.

For managed private project: `representative` is `verified` (confirmed working in Phase 0). The other five types are `not_yet_run` (commands exist but have not been run through OpsCenter yet).

## AUDIT_RUN_ID Injection

OpsCenter is the source of truth for `run_id`. The injection flow:

```
OpsCenter generates:  run_id = uuid4().hex
resolve_invocation_request sets:  env["AUDIT_RUN_ID"] = run_id
Phase 6 passes env to subprocess
VF reads: os.environ["AUDIT_RUN_ID"]
VF writes run_id into run_status.json
OpsCenter reads run_status.json and verifies run_id matches
```

The `ManagedAuditInvocationRequest` model validator rejects any request where `env["AUDIT_RUN_ID"]` is absent or does not match `run_id`.

## run_status.json Entry Point

The only permitted entry point for audit output discovery is `run_status.json`.

```python
from operations_center.audit_toolset import load_run_status_entrypoint

rs = load_run_status_entrypoint("/path/to/run_status.json")
```

`load_run_status_entrypoint(path)`:
- Raises `RunStatusNotFoundError` if the file does not exist
- Raises `RunStatusContractError` if the file fails Phase 2 Pydantic validation
- Returns `ManagedRunStatus` — a fully validated Phase 2 model
- Strips the `_example_note` key (used only in synthetic examples)
- Does NOT scan the directory for a run_status.json — the path must be explicit

The returned model exposes:
- `is_terminal` — whether the run has ended
- `is_compliant` — False for legacy `in_progress` files or missing `artifact_manifest_path`
- `has_manifest` — whether `artifact_manifest_path` is populated

## artifact_manifest_path Resolution

Once `run_status.json` is loaded, the manifest path is resolved exclusively from the `artifact_manifest_path` field:

```python
from operations_center.audit_toolset import resolve_artifact_manifest_path

manifest_path = resolve_artifact_manifest_path(rs, base_dir="/path/to/vf/repo")
```

`resolve_artifact_manifest_path(run_status, *, base_dir=None)`:
- Raises `ArtifactManifestPathMissingError` if `artifact_manifest_path` is None
- For relative paths: resolves against `base_dir` (managed repo root or run_status parent)
- Raises `ArtifactManifestPathResolutionError` if path is relative and `base_dir` is None
- For absolute paths: returns as-is
- Returns an absolute `Path` — the file is not read or validated at this stage

The `artifact_manifest_path` in managed private project run_status files is relative to the VF repo root (e.g. `tools/audit/report/representative/{bucket}/artifact_manifest.json`).

---

## No Directory Scanning Rule

OpsCenter must never scan directories to find manifests. The contract is:

```
run_status.json
  → artifact_manifest_path (explicit field)
    → artifact_manifest.json
```

There is no fallback. No glob. No directory listing. No path inference from bucket naming conventions.

If `artifact_manifest_path` is absent, `ArtifactManifestPathMissingError` is raised. The caller must surface this as a contract violation, not attempt to find the file another way.

This rule exists because:
1. OpsCenter must not know VF internals (bucket naming, directory shapes)
2. Discovery by scanning would silently accept stale or wrong manifests
3. Phase 5 must add `artifact_manifest_path` to VF — not knowing the path is a Phase 5 signal, not a Phase 3 workaround

## Error Types

All errors inherit from `ManagedAuditToolsetError`.

| Error | When raised | Failure kind |
|-------|-------------|-------------|
| `ManagedRepoNotFoundError` | No config file for repo_id | Configuration |
| `ManagedRepoCapabilityError` | Repo lacks 'audit' capability | Configuration |
| `ManagedAuditTypeUnsupportedError` | audit_type not in config | Configuration |
| `ManagedAuditCommandUnavailableError` | command_status is blocked | Command evidence |
| `RunStatusNotFoundError` | run_status.json file missing | Missing file |
| `RunStatusContractError` | run_status.json fails validation | Invalid contract |
| `ArtifactManifestPathMissingError` | artifact_manifest_path is None | Contract violation (Phase 5 gap) |
| `ArtifactManifestPathResolutionError` | Relative path with no base_dir | Caller error |

## Future CLI Shape

Phase 3 does not implement CLI commands. The expected future shape (Phase 6+):

```
operations-center audit prepare \
    --repo managed-private-project \
    --type representative \
    [--run-id <hex>]
    # → prints the ManagedAuditInvocationRequest (does not run)

operations-center audit inspect-run-status \
    --path tools/audit/report/representative/bucket/run_status.json
    # → validates and prints run_status summary

operations-center audit resolve-manifest \
    --path tools/audit/report/representative/bucket/run_status.json \
    --base-dir ../managed private project
    # → prints resolved artifact_manifest.json path
```

Thin read-only CLI wrappers may be added if consistent with the existing `entrypoints/` pattern. They must not execute audits, index artifacts, scan directories, or import managed private project.

---

## Non-Goals

- Phase 3 does not execute audit commands.
- Phase 3 does not write `run_status.json` or `artifact_manifest.json`.
- Phase 3 does not index artifacts or build artifact registries.
- Phase 3 does not implement dispatch orchestration (Phase 6).
- Phase 3 does not scan directories or infer paths from bucket naming.
- Phase 3 does not import managed private project Python code.
- Phase 3 does not add `artifact_manifest_path` to VF (Phase 5 job).
